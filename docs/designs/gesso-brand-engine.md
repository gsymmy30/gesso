## Landscape Context

### AGENTS.md Discovery (Eureka)
AGENTS.md is now a Linux Foundation standard with 60K+ repos. Adopted by Cursor, Copilot, Codex, Gemini CLI, VS Code, and more. The original design doc put agent instructions inside brand.md, but agents read AGENTS.md. This drove the Approach C decision: brand.md for humans, AGENTS.md for agents.

### Competitive Landscape
- No CLI tool exists that generates brand identity from code analysis
- All brand generators are web-based (Frontify, uBrand, Canva, Looka)
- None target developers specifically
- None analyze actual code to inform brand positioning
- The concept is genuinely novel

## Phasing

### Phase 1 (v1.0) — Core Generation
The "one command" experience. This is the launch.
- `npx gesso-cli` with repo analysis pipeline
- Optional 3-question user brief before analysis ("Who is this for?", "What problem?", "How different?") — skippable, falls back to code-only
- brand.json (canonical machine-readable spec) + brand.md (human-readable, rendered from JSON)
- AGENTS.md and CLAUDE.md brand snippets printed to terminal by default. `--write-agents` / `--write-claude` flags to mutate files with `<!-- gesso:brand-start/end -->` delimiters
- brand.tokens.json (CSS custom properties + Tailwind partial)
- Logo: text-based SVG wordmark with font glyphs converted to path outlines (font-independent)
- OG image: programmatic via `@vercel/og` (satori + resvg-js WASM, no native deps)
- Brand score (0-100) displayed before generation; projected score shown after
- Beautiful CLI output using generated brand colors (truecolor with ANSI 256 fallback via `chalk`)
- Interactive review mode (accept / regenerate / skip per section). Regenerating positioning cascades to voice + copy with warning.
- Diff mode (if brand.json exists, show colorized section diff, accept per-section)
- `.gessoignore` file support + auto-exclude secrets (.env, credentials.*, *.pem, *.key)
- `--dry-run` flag: show analysis result + what files would be sent to LLM
- `--no-color` flag for accessibility and piped output. Also respects `NO_COLOR` and `FORCE_COLOR` env vars (industry standard, chalk handles this automatically)
- `--verbose` / `--debug` flags for observability
- Prompt injection guard in all system prompts
- Schema version field in brand.md (`<!-- gesso:v1 -->`) and brand.json (`schemaVersion: 1`). On load, check version; if newer than current gesso, warn. If older, run migration function.
- Ctrl+C graceful save (accepted sections preserved)
- No-TTY auto-accept mode (for CI/piped usage)
- Regeneration cap: max 5 per section

## LLM Strategy

**Model:** Claude Sonnet (primary), OpenAI GPT-4o (fallback). All calls use structured output (Claude tool use / OpenAI JSON mode).

**Call architecture:** Two waves of parallel calls after a serial analysis step.
1. **Serial:** Repo read + analysis (single LLM call, ~4K input, structured output)
2. **Wave 1 (parallel, max 3 concurrent):** positioning, voice, copy, visual selection, SEO, agent instructions, claude instructions — 7 calls, ~1K output each
3. **Wave 2 (parallel, after Wave 1):** logo SVG rendering (programmatic, no LLM), OG image (programmatic via satori + sharp) — depends on colors from Wave 1

**Temperature:** 0 for analysis + archetype/palette selection (deterministic). 0.3 for copy generation (slight variation). 0 for voice rules and agent instructions (must be consistent).

**Repo sampling strategy:** Priority order: README.md > manifest files (package.json, Cargo.toml, pyproject.toml, go.mod) > src/ entry points > other src files > docs/ > tests (skip). Auto-skip: node_modules/, vendor/, dist/, build/, .git/, lockfiles (package-lock.json, yarn.lock, Cargo.lock), files > 50KB, binary files. Max 50 files, max 500 lines per file, total context cap of 16K tokens. Deterministic: sort by priority tier then alphabetically so re-runs produce identical sampling. Truncation: if over budget, drop lowest-priority files first.

**Token budget:** ~8-16K input for analysis call (repo context), ~1K output per generation call. Total: ~35K input, ~8K output. Cost: ~$0.05-0.15 with Claude Sonnet.

**Parallelization:** `p-limit(3)` for concurrent LLM calls. Exponential backoff on 429s (1s, 2s, 4s). Per-step timeout: 30 seconds.

**Time target:** ~45-60 seconds total (30s was aspirational, 45-60s is realistic with 8 LLM roundtrips at ~3-5s each parallelized in groups of 3).

**Core R&D risk:** The prompts that turn code signals into brand positioning are the hardest part. Plan for 2-3 prompt iteration cycles. Test on 10+ real repos of different types. The archetype library (curated palettes/fonts) constrains visual output quality; the prompts constrain text output quality.

## Brand Score Rubric

Score breakdown (0-100):

| Signal | Points | How |
|--------|--------|-----|
| Has a tagline/description in manifest | 10 | Check package.json description, Cargo.toml description, etc. |
| README has clear positioning (not just install instructions) | 15 | LLM judges: does README explain what this is and who it's for? |
| Consistent tone across docs | 15 | LLM compares tone of README vs other .md files |
| Has visual identity (logo, colors mentioned) | 10 | Grep for logo references, color values, brand assets |
| Has meta/OG tags | 10 | Check for og:image, og:description in any HTML/config |
| Feature naming follows a convention | 10 | LLM judges: are commands/features named consistently? |
| No banned words ("seamless", "powerful", etc.) | 10 | Grep for anti-pattern words in README and docs |
| Has an existing brand.md or similar | 10 | Check for brand.md, BRANDING.md, style-guide.md |
| Error messages have consistent voice | 10 | Sample error strings from source, LLM judges consistency |

Scoring: heuristic checks (grep-based) run first (fast, free), LLM checks run in a single batched call.

## Error Handling Strategy

**Fatal errors (pipeline aborts):**
- No repo found (invalid URL, path doesn't exist, private repo without GITHUB_TOKEN)
- No analyzable content (no README, no manifest, fewer than 3 source files)
- No LLM API key (neither ANTHROPIC_API_KEY nor OPENAI_API_KEY set)

**Recoverable errors (step skipped, pipeline continues):**
- Individual generation step fails after 2 retries or 30s timeout → section marked `[GENERATION FAILED — run gesso again or fill manually]` in brand.md
- Logo SVG validation fails → fallback to text-based wordmark in chosen font
- OG image rendering fails → omit og.png, annotate brand.md
- Rate limit (429) → exponential backoff (1s, 2s, 4s), then skip step

**User-facing errors:**
- Missing GITHUB_TOKEN for private repo: "This looks like a private repo. Set GITHUB_TOKEN to continue."
- Missing API key: "Set ANTHROPIC_API_KEY or OPENAI_API_KEY to use gesso."
- Repo too large (>100MB clone): "This repo is large. Gesso samples up to 50 files for analysis."

## Command Specifications

### `gesso check`
- **Input:** Reads brand.md from repo root. Scans README.md + all .md files in docs/ directory.
- **Checks against:** Voice tone words, Do/Don't rules, banned words list from brand.md
- **Output format:** Line-level annotations: `README.md:14 — "This powerful tool seamlessly..." → violates Don't list. Suggested: "Connects to your CI in one line."`
- **Scoring:** Pass/fail per file + overall voice consistency score (0-100)
- **LLM usage:** One LLM call per file being checked (batched if possible). The banned-words check is pure grep (no LLM needed).

### `gesso compare`
- **Input:** 1-3 public repo URLs. Runs full analysis pipeline on each.
- **Output:** Side-by-side table: archetype, tone words, color palette, positioning angle. Differentiation opportunities highlighted.
- **Limitation:** Public repos only. Brand signals come from code + README, not marketing sites. Set expectations accordingly.

### `gesso share`
- **Input:** brand.md from repo root. Requires GITHUB_TOKEN with gist scope.
- **Output:** Creates a GitHub Gist with brand.md content, returns URL. Gist description includes product name.

### Interactive Review Flow
After generation completes, sections are presented one at a time:
1. Positioning → accept / regenerate / skip
2. Voice & Tone → accept / regenerate / skip
3. Copy → accept / regenerate / skip
4. Visual Identity → accept / regenerate / skip
5. SEO → accept / regenerate / skip
6. Agent Instructions → accept / regenerate / skip
7. Logo preview (ASCII representation) → accept / regenerate / skip
"Regenerate" triggers a new LLM call for that section only (same context, temperature bumped by 0.1). "Skip" omits the section from output. No inline editing in v1 — user edits brand.md directly after generation.

### Diff Mode
When brand.json exists in repo root:
1. Generate new brand identity as normal
2. Show colorized section-by-section diff (using `diff` library)
3. Per section: accept new / keep old / skip
4. AGENTS.md and CLAUDE.md delimiters: `<!-- gesso:brand-start -->` / `<!-- gesso:brand-end -->`. Re-runs replace content between delimiters.

**Flag precedence with diff mode:** `--write-agents` and `--write-claude` are output-stage flags. They do not change the review flow. Diff mode runs first (user reviews section-by-section), then after review completes, the accepted result is written. If `--write-agents` is set, the AGENTS.md snippet from the final accepted brand is written using the delimiter strategy. Same for `--write-claude`.

### Error States
```
✗ No API key found.
  Set ANTHROPIC_API_KEY or OPENAI_API_KEY to use gesso.

✗ This looks like a private repo.
```

### Total Failure State
```
✗ Generation failed completely (0/8 steps succeeded).

  This usually means:
    • API key is invalid or lacks permissions
    • Network connection dropped during generation
    • Rate limit hit across all retries

  Try:
    gesso --debug       show full error details
    gesso --dry-run     verify analysis works before generation

  If this keeps happening, check your API key:
    echo $ANTHROPIC_API_KEY | head -c 10
```
When 0 steps succeed, skip interactive review entirely. Show actionable diagnostics, not a wall of errors. The `--debug` flag reveals full stack traces and API responses for troubleshooting.

### Partial Success State
```
✗ Generation partially failed (5/8 steps succeeded).

  ✓ Positioning, Voice & Tone, Visual Identity,
    Agent Instructions, Logo
  ✗ Copy (timeout), SEO (rate limited), OG Image (failed)

  Proceeding to review with completed sections.
  Failed sections marked [GENERATION FAILED] in brand.md.
  Run gesso again to retry failed sections (diff mode).
```
When some steps fail, still enter interactive review with what succeeded. Failed sections are skipped in review and marked in output.

## CLI Design Tokens

Consistent rules for all terminal output. Full design system is in DESIGN.md (source of truth).

**Spacing:**
- Indent: 2 spaces for all content under a header
- Section gap: 1 blank line between sections
- Stage gap: 2 blank lines between stages (score → brief → generation)

**Typography:**
- Section headers: `── {Name} ──────── [{N}/{Total}] ──` (em-dash + box-drawing)
- Stage headers: plain text, no decoration (e.g., "Generating brand identity...")
- Content: 2-space indent from left margin
- Max line width: 60 chars for wrapped content (fits 80-col terminals with indent)

**Gesso default palette (before brand colors load):**
Gesso's own brand identity. See DESIGN.md for full color system. Summary:
- Text: warm off-white (`chalk.hex('#E8E6E3')`) on dark terminals
- Headers: muted teal (`chalk.hex('#5BA4A4')`)
- Success: sage green (`chalk.hex('#7EC89B')`) for `✓`
- Error: terracotta red (`chalk.hex('#D4726A')`) for `✗`
- Warning/Accent: warm amber (`chalk.hex('#E8B87D')`)
- Muted: neutral gray (`chalk.hex('#8B8B8B')`) for secondary info like durations
- Progress spinner: muted teal (`chalk.hex('#5BA4A4')`) for `◐`
- Hint text: dim (`chalk.dim`) for "[a] accept [r] regenerate [s] skip"
- The moment brand colors load (after visual identity step), the spinner and headers visibly switch. This is an intentional "aha" moment.

**After brand colors load:**
- Spinner: primary brand color
- Section headers: accent brand color
- Score bars: primary brand color for filled, gray for empty
- Box borders: muted brand color

**Symbols:**
- Success: `✓` (U+2713)
- Failure: `✗` (U+2717)
- Spinner: `◐◓◑◒` cycle (U+25D0-25D3)
- Bar filled: `█` (U+2588)
- Bar empty: `░` (U+2591)
- Box: `┌ ┐ └ ┘ │ ─` (box-drawing characters)
- Section divider: `──` (U+2500)

**Fallback (`--no-color` or dumb terminal):**
- Success: `[OK]` instead of `✓`
- Failure: `[FAIL]` instead of `✗`
- Spinner: `-\|/` cycle
- Bar filled: `#`
- Bar empty: `.`
- No colors, no box-drawing, no Unicode symbols (plain ASCII only)
- Section headers: `-- {Name} -- [{N}/{Total}] --` (plain dashes)
- Box borders: `+`, `-`, `|` (ASCII box)
- This mode is also screen-reader friendly (no Unicode character names read aloud)

**Terminal width handling:**
- Minimum: 60 columns. If `process.stdout.columns < 60`, print warning: "Terminal is narrow (Xcols). Output may wrap. Resize to 60+ for best experience." Then continue (don't abort).
- Brand score bars: scale proportionally to available width. At 60 cols, bars are 10 chars. At 120 cols, bars are 20 chars.
- Section headers: fill available width with `─` characters.
- Content wrapping: wrap at `min(columns - 4, 76)` to maintain 2-char indent + 2-char margin.
- If `process.stdout.columns` is undefined (piped/CI): assume 80 columns.
